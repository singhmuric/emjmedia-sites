### Cluster C — Hero-h1 + Bento-Padding

- `.hero__h1` aktuell: `font-size: clamp(2.75rem, 2rem + 4.5vw, 4.5rem)` → @360px: 2rem + 4.5*3.6 = 18.2rem ?? Falsch — clamp middle-Argument ist `2rem + 4.5vw` = 32px + 4.5% der Viewport-Breite. @360 = 32 + 16.2 = 48.2 px (✅ deckt sich mit Diagnose 48.2 px). Untergrenze `2.75rem` = 44 px ist also irrelevant, weil middle@360 (48.2) > min (44). Fix: middle-Floor anziehen, dann greift es.
- Plan: `clamp(2.25rem, 1.5rem + 4.5vw, 4.5rem)` → @360 = 24 + 16.2 = 40.2 px. Bei 320 px = 24 + 14.4 = 38.4 px. Bei min greift `2.25rem` = 36 px nur für sehr schmale Viewports < ca. 320 px. @1440 = `min(24+64.8, 72) = 72` px → unverändert.
- Alternativ Handover-Vorschlag `clamp(2.25rem, 6vw, 4.5rem)`: @360 = max(36, 21.6) = 36 px. Aber middle ist niedriger, also greift unten oft 36 → das macht die Skala flacher als nötig. Mein Vorschlag oben hält die Skala intakt und senkt nur den Mobile-Range.
- `.leistungen` aktuell: `padding: var(--section-y) 0` — `--section-y` ist im tokens.css definiert (lese ich gleich). Mobile-Reduktion via `@media (max-width: 768px) { .leistungen { padding-block-start: clamp(2rem, 8vw, 3rem); } }`.

## 4. Mobile-Only-Strategie pro Cluster

| Cluster | Strategie | Begründung |
|---|---|---|
| A — Touch-Targets | `@media (pointer: coarse)` | Touch-Geräte adressieren, Desktop-Maus unberührt. Constitution §1.8 sagt Touch-Targets ≥ 44 — das ist eine Touch-Anforderung, nicht eine Maus-Anforderung. Auf Desktop-Hover-Geräten würden 44-px-Links visuell unschön „floppen". |
| B — Trust-Ribbon | `@media (max-width: 480px)` | Stack-Layout ist viewport-basiert (Schmale Viewports), nicht touch-basiert. iPad (768) bleibt waagrechte Ribbon. |
| C — Hero-h1 | `clamp()`-Floor senken (kein @media) | Senkt nur die Untergrenze — Desktop-Werte (mid + max) bleiben unverändert. |
| C — Bento-Padding | `@media (max-width: 768px)` | Padding-Reduktion trifft beide iPad-Hochformat und Mobile, lässt Desktop unangetastet. |
