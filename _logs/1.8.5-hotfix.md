## §0 — Pflicht-Diagnose ✅

### Diagnose-Befund
Tool: `scripts/oneoff/diagnose-header-185.mjs`
Report: `_logs/1.8.5-header-diagnose.md`

**Verdict: CONFIRMED — `.nav` hat semi-transparent bg + backdrop-blur.**

| Eigenschaft | Wert | Quelle |
|---|---|---|
| Element | `<header class="nav">` | DOM-Stack at viewport (720, 30) |
| `position` | `fixed` | computed-style |
| `background` | `rgba(11, 18, 32, 0.65)` | `01-nav.css:9` |
| `backdrop-filter` | `blur(8px)` | `01-nav.css:10` |
| `border-bottom` | `1px solid transparent` | `01-nav.css:12` |
| Pixel-Sample mid-bar (640×40) **vor Fix** | mean rgb(97.9, 101.6, 108.3) | viel heller als pure navy (11,18,32) |
| Δ Luminance vs pure navy | **+0.125** | bestätigt blur-mix mit hellem Foto darunter |

**Root-Cause des „weißen verschwommenen Effekts":** Der Header ist
semi-transparent (0.65 Nachtblau) und wendet `backdrop-filter: blur(8px)`
auf die Pixel darunter an. Das Werkstatt-Foto im Hero hat helle
Werkstatt-Decken-Beleuchtung — diese hellen Pixel werden durch den
Blur gemittelt und zur 0.65-opaken Nachtblau-Schicht hinzugemischt.
Effekt: Header erscheint hell-bläulich verwaschen.

Drei vorige Hotfix-Iterationen (1.8.2 / 1.8.3 / 1.8.4 §B) waren am
falschen Element (`.hero__media::before`) — sie haben das Werkstatt-
Foto im Hero-Container modifiziert, nicht den semi-transparenten
Header.

## §B — CTA Border ohne Doppellinie ✅

### Befund
1.8.3/1.8.4-Implementation hatte gleichzeitig:
- Statischer 1px-Messing-Border auf `.hero__cta--secondary`
- Animierter SVG-Hot-Spot, der über die Border-Kontur läuft

Bei kleinem Sub-Pixel-Versatz (selbst nach 1.8.4 dpr-aware Geometrie)
waren beide Linien gleichzeitig sichtbar → wirkte als Doppellinie
auf der rechten Vertikalen.

### Fix
- `styles/sections/03-hero.css`:
  - `.hero__cta--secondary { border: 1px solid transparent; }`
    Layout-Geometrie bleibt durch 1px-Reserve stabil, aber der
    statische Border ist visuell weg.
  - `@media (prefers-reduced-motion: reduce) { .hero__cta--secondary { border-color: var(--color-warm-accent); } }`
    Bei reduced-motion (animation aus) kommt der solide Messing-Border
    zurück — sonst wäre der Button konturlos auf dem Foto.
- `scripts.js` `updateCtaBorder()`:
  - dpr-aware Sub-Pixel-Snap: `Math.round(value * dpr) / dpr` für `width`
    und `height`. Auf Retina (dpr 2) → Geometrie auf 0.5-px-Raster
    statt nur ganze CSS-Pixel.

### Belege
- Frame-Captures `_logs/1.8.5-cta-frame-{0..4}.png` (normal motion):
  Während der Animation ist KEINE statische zweite Linie sichtbar,
  nur der wandernde Hot-Spot mit Glow.
- `_logs/1.8.5-cta-reduced-motion.png` (Puppeteer mit
  `emulateMediaFeatures prefers-reduced-motion: reduce`):
  Solider 1px-Messing-Border um den Button, KEIN Hot-Spot, KEIN Glow.
  Animation komplett aus.

### Acceptance §B
- [x] Während Animation: keine statische Messing-Linie sichtbar (Frames 0..4)
- [x] reduced-motion: solider Messing-Border + keine Hot-Spot-Animation
- [x] Rechte Vertikal-Seite: Hot-Spot nicht mehr neben einer statischen Linie
- [x] dpr-aware Math.round in updateCtaBorder
