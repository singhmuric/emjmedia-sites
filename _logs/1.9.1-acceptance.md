## 1. Diagnose-Befunde (vor Code-Änderung)

### FIX 1 — Hero-CTA „Termin anfragen" Mobile-Visibility

DOM-Inspect (`_logs/1.9.1-diagnose.json`):

- bbox `165,5 × 54,8 px` ab x=193,5 / y=496,7 (Hero im Viewport)
- `background-color: rgba(11, 18, 32, 0.95)` (95 % opaque Nachtblau)
- `border-color: rgba(0, 0, 0, 0)` + `border-width: 1px` (Border auf transparent gesetzt seit 1.8.5 §B)
- `backdrop-filter: blur(6px)` ✓
- SVG `.cta-border-loop` present, opacity 1, overflow visible
- rect: stroke `rgb(184, 146, 90)` (Messing), stroke-width `2.5px`, stroke-dasharray `62.46 / 353.94` (= 15 % / 85 % von Pfadlänge 416,4), animation `cta-march 3.2s linear infinite running` ✓
- JS-Init läuft (`updateCtaBorder('initial')` setzt `--cta-len` als rect-style; mein erstes Diagnose-Script las am falschen Element — `getComputedStyle(document.documentElement)` statt `getComputedStyle(rect)` — daher initial irrtümlich als „null" gemeldet, ist aber gesetzt).

Befund: **Animation und Background sind im Headless-Chrome korrekt gerendert.** Emins iPhone-Symptom („Kein Button-Body, animierte Border nicht erkennbar") ist iOS-Safari-spezifisch — vermutet: 95 % opakes Nachtblau verschmilzt visuell mit dem dunklen Werkstatt-Foto (brightness 0.78), `isolation: isolate` + `backdrop-filter` mag auf iOS render-Konflikte erzeugen.

**Fix 1 Plan:** `@media (max-width: 768px)` setzt
- `background: rgba(11, 18, 32, 0.55)` (Glas-Plate, ehrliche Translucenz)
- `backdrop-filter: blur(8px)` + `-webkit-backdrop-filter: blur(8px)` (Vendor-Prefix für iOS Safari)
- `border-color: rgba(184, 146, 90, 0.65)` (Messing als statischer Fallback-Border, falls SVG-Animation auf iOS unsichtbar)

Damit ist der Button-Bereich auf Mobile auch ohne SVG-Border klar vom Foto-BG getrennt. Desktop bleibt unangetastet (außerhalb @media).

### FIX 2 — Wartungsvertrag-Card Mobile-Layout

DOM-Inspect (gleicher Run):

- card bbox `343 × 718,8 px` (Höhe massiv aufgebläht!)
- **`grid-template-columns: 193,875px 73,125px`** ⚠ — die linke Spalte ist 194 px, die rechte nur 73 px
- icon `64 × 64` in cell(0,0) (linke Spalte)
- copy in cell(0,1) (rechte Spalte, 73 px)
- title bbox `73,1 × 69 px` (3-zeilig mit Mid-Word-Break, weil 73 px Breite zu schmal)
- cta bbox `193,9 × 47,1 px` in cell(1,0) (linke Spalte, zweite Reihe)

**Root-Cause:** Wartungsvertrag-Card hat 3 Children (icon · copy · cta), aber `grid-template-columns: auto 1fr` definiert nur 2 Spalten. CTA fließt in die 2. Reihe, linke Spalte. Implicit-Grid-Sizing nimmt die **breiteste** Cell jeder Spalte als Spalten-Breite — die CTA (`Beratung anfragen` als `btn-light` mit `min-height: 44px` und font-size 14) ist ~194 px breit → **zwingt die linke Spalte auf 194 px** → Copy-Spalte schrumpft auf 73 px → Title quetscht.

**Fix 2 Plan:** `@media (max-width: 768px)`:
- `.wartungsvertrag { grid-template-columns: 1fr; gap: 14px; }` → single-column-Stack: icon → copy → cta vertikal
- `.wartungsvertrag__icon { width: 48px; height: 48px; border-radius: 12px; }` (kleiner, damit oben nicht überdimensioniert)
- `.wartungsvertrag__title { font-size: 1.35rem; line-height: 1.2; text-wrap: balance; }` (balance zerlegt sauber, keine Mid-Word-Breaks)
- `.wartungsvertrag__text { margin-bottom: 16px; }`
- `.wartungsvertrag__cta { justify-self: stretch; width: 100%; }` (Block-CTA, full-width)
- `.wartungsvertrag::after { width: 100%; height: 50%; opacity: 0.6; }` (Overlay nicht mehr 40 % rechts, sondern Top-Drittel)

### FIX 3 — Sticky-Phone vs. Wartungsvertrag-CTA Overlap

DOM-Inspect:

- sticky-phone `position: fixed; bottom: 18px; right: 18px; z-index: 80;`, bbox `56 × 56`, `right: 357`, `bottom: 794` @ scroll-position
- visibility `is-shown: true`
- Bei aktueller Test-Scroll-Position keine direkte Überlappung mit `.wartungsvertrag__cta` (`overlapVsCta: ix=0, iy=0`)
- `pakete.bottom → sp.bottom delta = -402,8 px` (Pakete-Section bottom liegt 402 px unter sticky-phone bottom)

**Aber:** Nach Fix 2 wird die Wartungsvertrag-CTA full-width (`width: 100%`) → CTA right-edge bei `375 - 16 = 359 px`, sticky-phone left-edge bei `375 - 18 - 56 = 301 px`. **Horizontale Überlappung 58 px** an dem Scroll-Moment, in dem die Wartungsvertrag-CTA am Viewport-Bottom steht.

**Fix 3 Plan:** body padding-bottom auf Mobile, damit der unterste Content-Block nicht unter der schwebenden Sticky-Phone landet:
- in `14-sticky-phone.css`: `@media (max-width: 767px) { body { padding-bottom: clamp(72px, 10vh, 96px); } }`
- Sticky-Phone selbst bleibt unverändert — das ist der by-design free-floating Action-Button.

## 5. Lessons Learned (Skill-Log-Eintrag)

- **Cascade-Reihenfolge in Section-Partials:** Die erste 1.9.1-Iteration plazierte den `@media (max-width: 768px) { .hero__cta--secondary { ... } }`-Block direkt nach dem 1.9-`@media (max-width: 480px) { .hero__h1 { ... } }`-Block (Z. 140 in 03-hero.css). Der Default-`.hero__cta--secondary { background: rgba(11, 18, 32, 0.95) }` (Z. 187) kam **danach** und überschrieb cascade-mässig — Edit war wirkungslos. Fix: Mobile-Override-Blocks immer ans **absolute Ende** der Partial-Datei, nach allen Default-Declarations und Pseudo-/State-Selektoren. Bestätigt durch Re-Build + DOM-Inspect.
- **Template-Literal-Quoting im sed-Replace:** `replace_all` von `'-before.png'` zu `'-${phase}.png'` lässt die Single-Quotes intakt → `${phase}` ist literal im JS-String. Fix mit Backticks (sed-Korrektur). Lesson: bei dynamischen Pfaden in JS-Scripts immer Template-Literals (Backticks) nutzen, nicht Single-Quotes.
- **iOS-Safari Glas-Plate vs Solid:** 95 %-opaque dunkler Background auf dunklem Foto verschmilzt visuell — der Button verliert „Button-Charakter". Glas-Plate (rgba 0.55 + backdrop-blur) erzeugt klare Boundary-Trennung über die Blur-Kante, auch ohne harte Farb-Kontrast. Plus statischer Messing-Border als Sichtbarkeits-Fallback, falls SVG-Animation auf iOS-Render-Layer ausfällt.
