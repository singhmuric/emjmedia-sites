## §A — CTA Animated Border via SVG-Rect-stroke ✅

### Befund (vor Fix)
Conic-Gradient + mask-composite-Variante aus Session 1.8 §2.5 ließ
den Hot-Spot kreisförmig durch den Button-Innenraum wandern statt
auf der echten Rechteck-Kontur. Bei aspect-ratio !== 1 (Button ist
~200×55, Verhältnis 3.6:1) entsteht durch `transform: rotate()` auf
einem rechteckigen Element ein elliptischer Effekt im Innenraum.

### Fix
Conic-gradient + mask-composite raus. Ersetzt durch dediziertes
`<svg class="cta-border-loop">` mit einem `<rect>` über dem Button
und CSS-Animation auf `stroke-dashoffset`. Pfadlänge dynamisch
berechnet via `getTotalLength()`, an die Keyframes übergeben über
`--cta-len` Custom-Property. ResizeObserver hält die Pfadlänge bei
Resize aktuell.

### Belege
- HTML: `sites/onepages/kfz-demo/index.html` Zeile 185 — `<svg class="cta-border-loop"><rect rx="13" ry="13"/></svg>` innerhalb des `<a class="hero__cta--secondary">`
- CSS: `sites/onepages/kfz-demo/styles/sections/03-hero.css` Zeilen 99–166
  - `.hero__cta--secondary`: solider 1px-Messing-Border bleibt als Fallback
  - `.cta-border-loop`: `position:absolute; inset:-1px; pointer-events:none; z-index:1`
  - `.cta-border-loop rect`: `stroke-width:2; animation: cta-march 3.2s linear infinite`
  - `@keyframes cta-march`: `from { stroke-dashoffset: var(--cta-len, 200); } to { stroke-dashoffset: 0; }`
  - `prefers-reduced-motion`: animation aus, `.cta-border-loop` opacity:0, solider Border bleibt
- JS: `sites/onepages/kfz-demo/scripts.js` Zeilen 250–289 — `updateCtaBorder()` setzt viewBox/width/height/rect-Geometrie/dasharray/--cta-len; ResizeObserver-Hook auf den Button
- Rect-State (Live-Inspect, `_logs/1.8.1-cta-frame-0..4.png`):
  - Pfadlänge: 487.16 px
  - dasharray: `73.07 414.08` (15 % Hot-Spot, 85 % Gap)
  - rect width=200, height=55, rx=13 (responsive auf Button-Größe)
- Animations-Frames: Hot-Spot wandert sichtbar entlang der Border-Kontur:
  - frame-0 (t=0 ms): Hot-Spot unten links
  - frame-2 (t=1280 ms): Hot-Spot rechte Mitte
  - frame-4 (t=2560 ms): Hot-Spot oben rechts
  → entspricht ~80 % einer 3.2 s-Periode auf der Rechteck-Kontur, wie spezifiziert
- Subline-Kontrast unverändert: 11.67:1 (Re-Messung `_logs/1.8-hero-after-a.json`) — Hotfix A hat den 1.8 §2.2 Scrim-Fix nicht beeinträchtigt

### Acceptance §A
- [x] Hot-Spot sichtbar auf der echten Border-Kontur (siehe Frames)
- [x] Animation-Loop ca. 3,2 s (CSS animation duration, frame-Intervall verifiziert)
- [x] `prefers-reduced-motion`: animation aus, solider 1px-Messing-Border-Fallback bleibt sichtbar
- [x] Tote Conic-/Mask-Pfade entfernt (Constitution §13: keine toten CSS-Pfade)
- [x] Bei Resize: ResizeObserver triggert `updateCtaBorder()` neu — Pfadlänge wird angepasst, dasharray-Pattern bleibt 15/85 %

## §B — Top-Scrim Reichweite -60% 🟡 PAUSE / Eskalation

### Befund
Cowork-Review beschreibt einen "weißlich-hellen Nebel-Effekt am
oberen Hero-Rand" und bittet, dessen vertikale Reichweite um ~60 %
zu kürzen. Der Befund konnte im Code nicht reproduziert werden.

### Diagnose-Ergebnis
1. **CSS-grep**: in `sites/onepages/kfz-demo/styles/` existiert kein
   linearer Gradient mit weißem oder rgba(255, …, …, …) als
   Top-Fade. Die einzigen Gradients sind:
   - `.hero__scrim` (105deg/180deg, Nachtblau, Subline-Scrim 1.8 §2.2 — Constitution-relevant, nicht anfassen)
   - `.cta-border-loop` mask-Variante existiert seit Hotfix §A nicht mehr
   - `.hero__widget-divider` (90deg, Messing → transparent, dünner 1px-Strich in der Kennzahlen-Card)
2. **DOM-Inspect** (`scripts/oneoff/inspect-hero-top.mjs`,
   `_logs/1.8.1-hero-topstrip.png`): Element-Stack bei Hero-Top
   `(width/2, hero.y+12)` ist `IMG → PICTURE → DIV.hero__media →
   SECTION.hero → MAIN → BODY`. Kein zusätzliches Top-Scrim-Element
   im DOM.
3. **Pixel-Sampling** Hero-Top-Strip 80 px × 1440 px:
   gleichmäßig `rgb(53, 57, 67)` über alle 16 Sample-Reihen
   (y 0…75, 5px-Intervall). Das ist Nachtblau-Werkstatt-Foto durch
   `filter: brightness(0.78) contrast(1.04)`, nicht weißlich.

### Eskalation
Die Anweisung „Reichweite um ~60 % reduzieren" setzt einen
existierenden Top-Scrim-Gradient voraus. Der existiert nicht.
Mögliche Interpretationen, die User-Klärung brauchen:

a) **Cowork sieht den hellen Werkstatt-Decken-Bereich des Fotos
   selbst** (object-position: center 35% legt obere Bildkante in
   einen helleren Werkstatt-Bereich). Lösung wäre dann
   `object-position` auf `center 50%` zu schieben oder
   `filter: brightness(0.78)` zu reduzieren — beides eine bewusste
   Foto-Anpassung, keine Scrim-Änderung.

b) **Cowork hat einen Top-Scrim erwartet, der nie eingebaut wurde**
   und möchte ihn jetzt mit reduzierter Reichweite haben. Das wäre
   eine neue Section/Layer, also Scope-Erweiterung — laut Hotfix-
   Ampel 🔴 nicht ohne explizite Spec-Freigabe.

c) **Falsche Site-Version reviewed** (Cowork hat ggf. eine ältere
   1.6/1.7-Variante oder eine andere Branche im Kopf). Der KFZ-Demo
   hatte in 1.6, 1.7, 1.8 keinen weißen Top-Scrim.

**Empfehlung:** Bevor irgendeine CSS/HTML-Änderung am Hero passiert,
Cowork bitte einen Pixel-Bereich (Screenshot mit Markierung oder
Koordinaten) angeben, der den „weißlichen Nebel" zeigt — gegen den
Live-Stand aus 1.8 (`_logs/1.8-hero-after-a.png`). Dann ist klar,
ob es sich um (a), (b) oder (c) handelt.
