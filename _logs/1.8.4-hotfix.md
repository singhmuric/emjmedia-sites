## §A — CTA Border Sub-Pixel-Stabilität ✅

### Befund (vor Fix)
Hot-Spot lief an oberen, unteren und linken Kanten sauber, aber an der
**rechten Vertikalen** war im Screenshot ein Sub-Pixel-Versatz zu sehen.
Vermutete Root-Cause: `updateBorder()` lief bei DOMContentLoaded mit
einer Button-Bbox, die noch nicht die finalen Web-Font-Metriken hatte
(Fraunces lädt async nach DOM-Parse).

### Fix
1. **Mehrfach-Trigger** in `scripts.js` `updateCtaBorder()`:
   - `'initial'` bei Module-Init (DOMContentLoaded-äquivalent in der IIFE)
   - `'load'` via `window.addEventListener('load', …)`
   - `'fonts.ready'` via `document.fonts.ready.then(…)` (nur wenn FontFace-API da)
   - `'resize'` via bestehenden ResizeObserver
   → Geometrie wird damit mind. 4× nach allen Layout-Settling-Events neu berechnet.

2. **Math.round() auf bbox** (vorher implizit float):
   - `w = Math.round(bbox.width + 6)`
   - `h = Math.round(bbox.height + 6)`

3. **Sub-Pixel-Stabilität via GPU-Layer:**
   - `transform: translateZ(0)` auf `.cta-border-loop` → eigener Compositor-Layer, stabilere Sub-Pixel-Rendering ohne Flackern.

4. **Inset von -2 px auf -3 px** (mehr Reserve nach außen):
   - SVG `inset: -3px` + `width/height: calc(100% + 6px)`
   - `bbox + 6` statt `bbox + 4` → 1.5 px Stroke-Mittellinie + 1 px Reserve außen + 0.5 px für Sub-Pixel-Drift.

5. **Diagnose-Log** (opt-in, nur wenn `window.__ctaDiag = true`):
   - Pro Trigger ein Eintrag mit `{ timestamp, trigger, bboxW, bboxH }` in `window.__ctaDiagLog`
   - Test liest aus → `_logs/1.8.4-cta-geometry.txt`

### Belege
- CSS: `styles/sections/03-hero.css` Zeilen 116–129 (`.cta-border-loop` mit `transform: translateZ(0)` + `inset: -3px`)
- JS: `scripts.js` Zeilen 250–308 (updateCtaBorder + diagLog + 4 Trigger-Hooks)
- HTML: `index.html` Zeile 185 (rx=14 unverändert seit 1.8.3)

### Diagnose-Log (`_logs/1.8.4-cta-geometry.txt`)
```
2026-04-25T14:09:23.951Z | bbox=199.71x54.75 | trigger=initial
2026-04-25T14:09:23.958Z | bbox=199.71x54.75 | trigger=resize
2026-04-25T14:09:23.968Z | bbox=199.71x54.75 | trigger=load
2026-04-25T14:09:23.981Z | bbox=199.71x54.75 | trigger=fonts.ready
```
→ alle 4 Trigger feuern. In dieser Page-Load lieferte fonts.ready
keine Bbox-Änderung mehr (Fonts waren beim DOMContentLoaded bereits
durch network-idle vorgeladen). Aber: defense-in-depth bei langsamerer
Verbindung greifen die zusätzlichen Trigger automatisch.

## §B — Top-Bar opaque + Messing-Trennlinie ✅

### Strategiewechsel
Schleier-Variante (1.8.2 + 1.8.3) wird **ersetzt**. Cowork-Vorgabe:
- Oberen Bereich vollständig verdecken (LEDs unsichtbar)
- 1px-Messing-Linie als Designs-Akzent

### Fix
- `styles/sections/03-hero.css` Zeile 21–48 (`.hero__media::before`):
  ```css
  .hero__media::before {
    content: "";
    position: absolute;
    inset: 0 0 auto 0;
    height: 16%;
    background: rgb(11, 18, 32);                       /* opak */
    border-bottom: 1px solid rgba(184, 146, 90, 0.55); /* Messing-Trennlinie */
    pointer-events: none;
    z-index: 1;
  }
  ```
- KEIN `backdrop-filter` (Performance-Schutz, 🟡-Ampel)
- `.hero__scrim` (Subline-Scrim 1.8 §2.2) unangetastet (Constitution §13)

- mean rgb(11.0, 18.0, 32.0)

- → PASS voll opak, exakt wie spezifiziert
